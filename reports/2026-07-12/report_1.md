## Ubuntu の「snap」を試す
- Date: 2026-07-11T08:10:41+00:00

### Executive Summary
- Ubuntu 24.04 環境で snap の基本操作を試したスクラップ。

- snapd がインストール、隔離、更新処理を管理する点が中心に説明されている。
- Markdown 関連パッケージ検索から glow の詳細確認、インストール、実行までを実演している。
- snap パッケージは /snap 配下に配置され、/snap/bin から実行されることを確認している。
- 更新は自動 refresh が基本で、hold と unhold により一時停止や再開が可能である。
- vlc を例にチャネル変更、refresh、revert、revision 指定の挙動を確認している。
- 隔離機能や connections によるアクセス権の確認にも触れており、snap の運用面を広く確認している。

### Key Findings

  - Footnote: 記事では「snap パッケージと snapd は、基盤となる Linux ディストリビューションやハードウェア環境に依存せず」と説明している。

  - Footnote: 実行結果として snap 2.75.2+ubuntu24.04、snapd 2.75.2+ubuntu24.04、ubuntu 24.04 が表示されている。

## OpenAI Responses API の Programmatic Tool Calling
- Date: 2026-07-10T17:34:54+00:00

### Executive Summary
- OpenAI Responses API の Programmatic Tool Calling を公式ドキュメントベースで整理したメモ。
- Responses API リクエスト内で JavaScript コードを生成・実行し、ツール呼び出しを連携できる機能として説明している。
- 並列呼び出し、ループ、条件分岐、中間結果保持など、従来の直接的 Tool Calling より制御フローをコード化しやすい。
- 生成コードは毎回新しい隔離 V8 ランタイムで実行され、Node.js や直接ネットワークアクセスは提供されない。
- 外部システムとのやり取りは、リクエストで有効化されたツール経由に限定される。
- 単一呼び出しや承認が必要な操作では、従来の直接的 Tool Calling が推奨される。
- 結果の絞り込み、結合、重複除去、集計、検証など、予測可能な処理には Programmatic Tool Calling が向いている。
- ZDR ワークフローとの関係にも触れ、store: false だけでは ZDR 有効化にならない点を注意している。

### Key Findings
- Programmatic Tool Calling は Responses API 内で JavaScript によるツール連携を可能にする。 [^]
  - Footnote: 記事では「Responses API リクエスト内でツール呼び出しを連携・実行するための JavaScript コードを生成・実行」と説明している。
- 並列処理や条件分岐を含む制御フローをコード側で扱える。 [^]
  - Footnote: できることとして「プログラムからツールを並列呼び出し」「ループ・条件分岐を使用」「中間結果を保持」が列挙されている。
- 実行環境は隔離された V8 ランタイムで、永続的な JavaScript 状態はない。 [^]
  - Footnote: 記事では「隔離されたV8ランタイム環境が毎回新しく用意」され、「プログラム実行間で持続するJavaScriptの状態」は提供されないとしている。
- Node.js、パッケージインストール、直接ネットワーク、ファイルシステム、サブプロセス、コンソール出力は使えない。 [^]
  - Footnote: 提供されないものとして Node.js 環境、パッケージのインストール、直接的なネットワークアクセス、汎用的なファイルシステム、サブプロセス実行機能、コンソール出力機能が列挙されている。
- 外部システム連携は有効化されたツール経由に限定される。 [^]
  - Footnote: 記事では「プログラムが外部システムとやり取りできるのは、リクエストで有効化されたツール経由のみ」と記載している。
- 予測可能な制御フローや構造化結果の生成には Programmatic Tool Calling が推奨される。 [^]
  - Footnote: 推奨条件として「処理のある段階で制御フローが予測可能な場合」「コードによって、より小さく構造化された結果を返すことができる場合」が挙げられている。
- 承認が重要な操作やモデル判断が逐次必要な操作は直接的 Tool Calling が原則となる。 [^]
  - Footnote: 記事では「各結果に対して新たなモデル判断が必要な場合」「作業に承認が必要な場合」は直接的ツール呼び出しを使用するとしている。
