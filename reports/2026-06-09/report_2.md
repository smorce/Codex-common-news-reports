## AI駆動開発を2コマンドで組織標準に ── Claude Code × Codexで設計からテストまで - ZOZO TECH BLOG
- Date: 2026-06-08T00:00:00+09:00

### Executive Summary
- ZOZO基幹システム本部がAI駆動開発を2つの標準コマンドに集約した事例である。
- 背景には、Claude CodeやCodexの使い方が個人ごとに閉じ、再現性が下がる問題がある。
- 導入済みツールにはClaude Code、Codex、Devin、Cursorなどがあり、Claude Codeは数百名規模で使われている。
- 方針は、Jiraから設計書や進捗管理表を作る初回コマンド/dev-initと、文脈を失わず再開する/dev-resumeに分けることだ。
- AIには具体から抽象、同一レベル変換を任せ、企画や優先順位など抽象から具体への判断は人間が主導する。
- Claude Codeが生成と採否判断を担い、Codexが独立調査に基づく批判的レビューを担当する。
- フロントエンド変更ではPlaywright MCPも接続し、画面確認まで標準化する構想になっている。

### Key Findings
- 課題はAI未利用ではなく、AI活用のプロンプト、判断基準、レビュー観点が個人差に依存することだった。 [^]
  - Footnote: 記事は「問題は『AIを使っていないこと』ではなく『使い方が個人に閉じていること』」と述べている。
- ZOZOは開発AIエージェントを全エンジニアへ導入する方針を発表している。 [^]
  - Footnote: 本文に2025年7月、1人あたり月額200ドルを基準として全エンジニアに導入するとある。
- 2コマンド化は、入口を増やしすぎず、初回状態作成と作業再開を分離するための設計である。 [^]
  - Footnote: 記事は/dev-initを初回作成、/dev-resumeを再開に分けた理由を説明している。
- AIに任せる範囲は、情報変換の方向で整理されている。 [^]
  - Footnote: 本文は「具体から抽象」「同一レベルの変換」「抽象から具体」の3分類で線引きしている。
- Claude CodeとCodexを組み合わせ、単一モデル依存やセルフレビューの弱さを補っている。 [^]
  - Footnote: 記事はClaude Codeのみでは単一モデルの見落としや性能調整の影響を受けやすいと説明する。
- 進捗管理表をGit管理できる文脈ファイルにし、会話履歴に依存しない再開を可能にしている。 [^]
  - Footnote: 本文に、文脈を会話ではなくファイルに持たせ、AIと人間が同じ状態を参照するとある。
