## 「DiffusionGemma」を試す

### Executive Summary
- Google の DiffusionGemma を試すためのスクラップで、公式ブログやモデルカードの内容を日本語で整理している。
- DiffusionGemma は通常の自己回帰 LLM と異なり、256 トークン程度のブロックを並列に生成し、反復的に整える拡散型のテキスト生成モデルとして説明されている。
- 主な価値はローカル GPU や低バッチ環境での高速応答で、H100 や RTX 5090 で高い tokens/sec が示されている。
- モデルは 26B 規模の MoE だが、推論時の有効パラメータは 3.8B 程度で、量子化により 18GB VRAM クラスでも扱える可能性がある。
- 双方向注意により、コード補完、インライン編集、構造を持つテキストやパズルのような非線形タスクとの相性が強調されている。
- 一方で、生成品質は通常の Gemma 4 より低いとされ、最高品質が必要な本番用途では自己回帰モデルが推奨される。
- Hugging Face、Unsloth、MLX、vLLM、Transformers、NVIDIA 系ツールなど、利用経路とエコシステムも整理されている。

### Key Findings
- DiffusionGemma はテキスト生成向けの実験的なオープンソースモデルとして位置づけられている。 [^]
  - Footnote: 本文に「Apache 2.0ライセンスの下でリリースされた、テキスト生成の高速アプローチを探求する実験的なオープンソースモデル」とある。
- 逐次生成ではなく、テキストブロックを同時生成することが基本的な違いである。 [^]
  - Footnote: 本文に「テキスト全体のブロックを同時に生成します」「1トークンずつじゃなくて 256トークンまとめて同時生成」とある。
- 高速化の主張は GPU 上の低レイテンシ用途に重点がある。 [^]
  - Footnote: 本文に「最大4倍の速度向上」「単一のNVIDIA H100で1秒あたり1000トークン以上、NVIDIA GeForce RTX 5090で1秒あたり700トークン以上」とある。
- MoE 構成により、総規模に対して推論時の有効パラメータを抑えている。 [^]
  - Footnote: 本文に「推論時に3.8Bパラメータのみを活性化する26BのMixture of Experts (MoE)モデル」とある。
- 双方向注意により、前後関係を同時に見た編集・補完タスクが期待されている。 [^]
  - Footnote: 本文に「各トークンが他のすべてのトークンに注目できます」「インライン編集、コード補完、数学的グラフなどの非線形ドメイン」とある。
- 生成品質には明確なトレードオフがある。 [^]
  - Footnote: 本文に「品質は普通の Gemma 4 より落ちる」「プロダクションで最高品質が絶対条件なら、ふつうのオートレグレッシブ Gemma 4 推奨」とある。
- 長文・マルチモーダル・関数呼び出しなど、モデルカード由来の機能も整理されている。 [^]
  - Footnote: 本文に「最大256,000トークンのコンテキストウィンドウ」「画像理解」「動画理解」「関数呼び出し」とある。

## Herdr

### Executive Summary
- Herdr は、複数のコーディングエージェントを 1 つのターミナルで管理するためのツールとして紹介されている。
- 筆者は Zed のエージェント評判を調べる中で Herdr を見つけ、関連する X 投稿と公式サイトの説明を記録している。
- Herdr は tmux がターミナルにもたらすような多重化体験を、コーディングエージェント向けに提供するものと説明されている。
- エージェントが実行されるマシンや SSH 接続可能なサーバー上で動作する点が特徴として挙げられている。
- セッション保持、SSH 経由での利用、速度、バグの少なさ、フォークしやすさが比較上の利点として言及されている。
- 公式サイト説明では、ブロック中・作業中・完了などの状態を一目で確認できることが強調されている。
- スクラップは短いが、ターミナル中心のエージェント運用を改善するツールとしての関心が読み取れる。

### Key Findings
- Herdr はコーディングエージェントの管理をターミナル内で行うツールとして扱われている。 [^]
  - Footnote: 本文に「1つのターミナルで全エージェントを管理」「エージェント多重化 · ターミナル内で動作」とある。
- Herdr の設計思想は tmux 的な多重化に例えられている。 [^]
  - Footnote: 本文に「Herdr はコーディングエージェントにとって tmux がターミナルにもたらすものと同様の機能を提供します」とある。
- エージェントが動く環境に近い場所で動作する点が特徴である。 [^]

## 「Unlimited-OCR」を試す

### Executive Summary
- Baidu の Unlimited-OCR を試すスクラップで、モデルの概要、公開情報、Colaboratory L4 での実行メモがまとめられている。
- Unlimited-OCR は長文書を一度に読むことを目的にした OCR モデルとして紹介され、OmniDocBench での SOTA 達成が言及されている。
- モデルは総パラメータ 3B、アクティブ 500M と説明され、長文書処理を効率化する R-SWA が主要な技術として挙げられている。
- R-SWA はソース、直近文脈、次の単語に注目し、不要な情報を柔らかく忘れる仕組みとして説明されている。
- 一定の KV Cache サイズと低いアテンションコストにより、40 ページ以上を単一フォワードパスで書き写せるとされている。
- Colaboratory では Transformers 5 系では動かず、4 系へのダウングレードと addict、pymupdf の追加が必要だったと記録されている。
- 単一画像と複数画像の推論例が示され、PDF は PyMuPDF で画像化してから処理する流れが紹介されている。

### Key Findings
- Unlimited-OCR は長文書を一度に読むための OCR モデルとして公開された。 [^]
  - Footnote: 本文に「Unlimited OCRをオープンソース化します — 長文書を一度に読むために構築されたもの」とある。
- モデル規模は総 3B、アクティブ 500M とされている。 [^]
  - Footnote: 本文に「総パラメータ3B、アクティブ化されたものはわずか500M」とある。
- OmniDocBench v1.5 および v1.6 でエンドツーエンドの SOTA 結果を達成したと紹介されている。 [^]
  - Footnote: 本文に「OmniDocBench v1.5およびv1.6で新たなエンドツーエンドのSOTA結果を達成」とある。
- 主要技術は Reference Sliding Window Attention である。 [^]
  - Footnote: 本文に「鍵となる革新はReference Sliding Window Attention (R-SWA)です」とある。
- R-SWA は長文書処理時に文脈とコストの両立を狙う仕組みとして説明されている。 [^]
  - Footnote: 本文に「ソース、最近の文脈、次の単語に焦点を当てつつ、不要になったものを柔らかく忘れる」とある。
- 40 ページ以上の文書を単一フォワードパスで扱える点が強調されている。 [^]
  - Footnote: 本文に「単一のフォワードパスで40ページ以上の書き写しが可能 — 文脈を失うことなく、速度を落とすこともありません」とある。
- Colaboratory 実行では依存関係の調整が必要だった。 [^]
  - Footnote: 本文に「transformers は5系だと動かない」「!pip install "transformers>=4,<5"」「!pip install addict pymupdf」とある。
- L4 環境でモデルロード後の VRAM 消費は約 6.7GB と記録されている。 [^]
  - Footnote: 本文に「この時点でのVRAM消費は6.7GB程度」とあり、NVIDIA-SMI 出力にも 6684MiB とある。
