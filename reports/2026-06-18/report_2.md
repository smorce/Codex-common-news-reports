## 【完全ローカル】AIに記憶を持たせる5ステップ — Ollama×RAGでつくる長期記憶チャットボット #rag - Qiita
- Date: 2026-06-15T00:00:00+09:00

### Executive Summary
- ローカル環境だけで長期記憶付きチャットボットを作る実装手順を解説している。
- 最初に全会話履歴を LLM に渡す素朴な方式を示し、トークン増加と精度低下の問題を確認している。
- RAG を導入し、質問に関連する過去の会話だけを検索してモデルへ渡す構成に置き換えている。
- ChromaDB と埋め込みモデルを使い、会話履歴をベクトル検索できる形で保存する。
- 生ログをそのまま保存せず、後で役立つ重要な事実だけを抽出するメモリ設計を採用している。
- 重複検出、クエリリファクタリング、タイムスタンプによる新旧判定を段階的に追加している。
- Ollama、llama3、nomic-embed-text、Python、ChromaDB を使うため API キーなしで動かせる。
- 記事は RAG がモデルの知能そのものを補うものではなく、記憶参照の仕組みである点も注意している。

### Key Findings
- 全履歴を毎回 LLM に渡す方式は、会話が長くなるほどトークン数、費用、遅延、精度劣化の問題を起こす。 [^]
  - Footnote: 記事は「会話が長くなるほど、毎回すべての履歴をLLMに渡すことになる」とし、トークン上限、コスト増、精度低下、レスポンス遅延を列挙している。
- RAG の中心は、全部を渡さず、現在の質問に関係する情報だけを取り出して渡すことにある。 [^]
  - Footnote: 記事は RAG の核心を「全部渡すのではなく、今の質問に関係する情報だけを取り出して渡す」と説明している。
- 基本構成では、質問を埋め込みに変換し、ChromaDB で類似検索して上位の過去会話を取得する。 [^]
  - Footnote: 記事は「新しい質問→埋め込みベクトルに変換→ChromaDBで類似検索→上位N件の過去の会話を取得」という流れを示している。
- 検索で取り出した断片は会話ターンではなく、参考情報としてシステムプロンプトに入れる設計が推奨されている。 [^]
  - Footnote: 記事は、検索断片を assistant/user のターンとして並べると前後関係が壊れ、モデルが混乱すると説明している。
- 生ログ保存ではノイズが増えるため、長期記憶には重要な事実だけを抽出して保存する方式が有効とされる。 [^]
  - Footnote: 記事は、あいさつや相槌まで保存すると検索枠を消費し、埋め込みの要点もぼやけると述べている。
- 実装はすべてローカルで動作し、Ollama、llama3、nomic-embed-text、ChromaDB、Python 3.10+ を使う。 [^]
  - Footnote: 記事の技術スタック表には、LLM 実行が Ollama + llama3、埋め込み生成が nomic-embed-text、ベクトル DB が ChromaDB と記載されている。
